# 功能概述
将声明式 UI 中的内置组件调用（`Text('hello')`、`Button('click')` 等）转换为命令式 `Component.create(args)` / `Component.pop()` 模式，同时处理自定义组件实例化和特殊组件的定制转换。

| 转换规则 | 说明 |
|---|---|
| 内置组件 create/pop | `Text('hello')` -> `Text.create('hello') ... Text.pop()` |
| 自定义组件实例化 | `MyComp({param: val})` -> `new MyComp(parent, params, ...)` 或 `_invokeImpl(...)` |
| Partial Update 模式包装 | Partial Update 模式下组件创建包裹在 `observeComponentCreation2` 回调中 |
| 特殊组件处理 | 部分组件有定制转换逻辑，详见下方"特殊组件处理"章节 |

## 动静态差异说明

| 维度 | 动态工具链 | 静态工具链 |
|---|---|---|
| 内置组件 | 生成 `Component.create(args)` / `Component.pop()` 命令式序列 | 组件调用通过 builder lambda 链式展开，不生成 create/pop |
| 自定义组件 | `new MyComponent(parent, params, ...)` 实例化 | `MyComp._invokeImpl(style, initializers, ...)` 静态方法调用 |
| Partial Update 包装 | `observeComponentCreation2(arrowFn, componentName)` | 无此包装 |
| observe 选择 | 4 种 observe 方法（Recycle/V2/PU2/普通） | `_invokeImpl` 统一入口 |
| StartGet/StopGet | 显式包裹（Partial Update V2 优化跳过） | 由 builder-lambda 链处理 |
| 组件定义来源 | `components/*.json`（145 个）+ `component_map.ts` | `predefines.ts` 枚举 + `arkuiImportList.ts`（165 个） |

## 接口声明交叉验证

### 声明文件格式差异

`arkui/component/` 目录下只有 `.static.d.ets` 文件，无 `.d.ets` 动态版本。动态工具链使用 `components/*.json` 作为组件注册表，不直接消费 `.static.d.ets`；静态工具链直接消费 `.static.d.ets`。部分组件标记 `@noninterop` 表示不参与 interop 桥接，静态工具链通过 `arkuiImportList.ts`（165 个条目）识别并跳过。

### ExtendableXxx 与 XxxModifier

SDK 声明定义 `ExtendableXxx` 类（如 `ExtendableButton`、`ExtendableText`），基类为 `ExtendableComponent`（`predefines.ts:608`）。静态工具链通过 `BuilderLambdaNames` 枚举管理方法名：`$_instantiate` → `_instantiateImpl`，`$_invoke` → `_invoke`。

| 维度 | ExtendableXxx | XxxModifier |
|---|---|---|
| 用途 | 声明式链式属性调用 | 命令式动态属性修改 |
| 调用方式 | `Text('hello').fontSize(20).fontColor(Color.Red)` | `new TextModifier().fontSize(20)` |
| 使用场景 | build() 方法内声明式 UI | 运行时动态修改 |
| 工具链处理 | `bindComponentAttr` / `builder-lambda-translators` 链式展开 | `isAttributeModifier` 分支 |

两者实现相同的 `XxxAttribute` 接口，确保 API 一致性：`ExtendableXxx` 用于编译时声明式设置（可优化），`XxxModifier` 用于运行时动态修改（灵活）。

### ListItem和GridItem

`ListItem`/`GridItem`/`ArcListItem` 在 Partial Update 模式下有特殊的懒创建（lazy create）优化处理：LazyForEach 本身已做懒加载，其直接子 ListItem 不需要再套一层 lazy create 机制（避免双重懒加载）。

#### 源码参考位置
- `compiler/src/process_component_build.ts:948`（`processInnerComponent` 中 `ItemComponents` 判断入口）
- `compiler/src/process_component_build.ts:1334`（`processItemComponent`）
- `compiler/src/process_component_build.ts:1449`（`checkLazyCreate`，懒创建判定）
- `compiler/src/process_component_build.ts:3770`（`isLazyForEachChild`）
- `compiler/src/pre_define.ts:622`（`ItemComponents = ['ListItem', 'GridItem', 'ArcListItem']`）

#### 转换前的原始代码
```typescript
// 场景1：ListItem 在 LazyForEach 内
LazyForEach(this.dataSource, (item) => {
  ListItem() { Text(item) }
})

// 场景2：ListItem 在普通 List 内

  ListItem() { Text('item1') }
}
```

#### 转换后的代码（Partial Update）
```typescript
// 场景1：LazyForEach 内 - 不启用 lazy create
LazyForEach.create(id, this, this.dataSource, (item) => {
  this.observeComponentCreation2((elmtId, isInitialRender) => {
    ListItem.create()
    if (!isInitialRender) { ListItem.pop() }
  }, ListItem)
}, ...)

// 场景2：普通 List 内 - 启用 lazy create
List.create()
ListItem.create()
this.observeComponentCreation2((itemCreation2, ListItem) => {

}, ListItem)

```

#### 关键转换逻辑
- `processInnerComponent`（line 948）：Partial Update 模式下 `ItemComponents.includes(name)` 为 true 时调用 `processItemComponent`（而非 `processNormalComponent`）
- `checkLazyCreate`（line 1449-1464）判定是否启用 lazy create，有三条短路条件返回 false：
  - 第一个参数是字符串字面量 `'false'`（开发者显式禁用）

- lazy create 分支生成 `itemCreation` 箭头函数（含 `StartGetAccessRecordingFor`/`StopGetAccessRecording` 包裹）和 `deepRenderFunction`（用于深度渲染）

### 4 种 observe 方法选择逻辑

| 条件 | observe 方法 | 说明 |

| 其他 | `observeComponentCreation` | 非 Partial Update 优化模式 |

### observe 函数调用参数对比

| observe 方法 | 参数结构 |
|---|---|

| `reuseOrCreateNewComponent` | `[{ componentClass, reuseId, ... }]` |

- `parentConditionalExpression`（line 1995-2002）：全局 Builder 场景生成 `parent ? parent : this` 三元，非全局 Builder 使用 `this`

### JSON attrs 与 SDK Attribute 接口对照

6 个代表性组件（Text、Button、Column、TextInput、Slider、Toggle）的 JSON `attrs` 数组与 SDK 声明文件中的 `XxxAttribute` 接口方法签名对照。

#### 对照结果

| 组件 | JSON attrs 来源 | SDK 声明接口 | 匹配度 |
|---|---|---|---|
| Text | `text.json` attrs | `TextAttribute extends CommonAttribute` | 100%（attrs 全部在接口中找到同名方法） |
| Button | `button.json` attrs | `ButtonAttribute extends CommonAttribute` | 100% |
| Column | `column.json` attrs | `ColumnAttribute extends CommonAttribute` | 100% |
| TextInput | `textInput.json` attrs | `TextInputAttribute extends CommonAttribute` | 100% |
| Slider | `slider.json` attrs | `SliderAttribute extends CommonAttribute` | 100% |
| Toggle | `toggle.json` attrs | `ToggleAttribute extends CommonAttribute` | 100% |

#### 通用发现

1. JSON `attrs` 数组中的每个属性名都能在对应 SDK 声明文件的 `XxxAttribute` 接口中找到同名方法 — **完全匹配**。
2. SDK 声明文件比 JSON 更丰富：`XxxAttribute` 接口包含 JSON 中没有的通用方法（如 `attributeModifier(modifier: AttributeModifier<XxxAttribute>)`），这些通用方法对所有组件都适用，不需要在每个组件 JSON 中重复，由 `common_attrs.json` 集中管理。
3. JSON 是动态工具链的验证数据层（`component_map.ts` 运行时加载），只需包含组件特有属性；SDK 声明是静态工具链的权威来源，包含完整类型信息。
