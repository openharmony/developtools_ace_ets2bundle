# 规格文档打分模型

## 适用场景

当需要对 `docs/specs/` 目录下的规格文档进行质量评估时，使用本模型对每个文档打分，识别低质量文档并指导优化。

## 打分维度与权重

总分 100 分，由 7 个维度加权构成：

| 维度 | 权重 | 满分 | 评估重点 |
|---|---|---|---|
| 源码准确性 | 25% | 25 | 源码参考位置是否真实存在且行号准确 |
| 内容完整性 | 20% | 20 | 是否覆盖该主题的所有关键转换/校验规则 |
| 代码示例质量 | 20% | 20 | 转换前/后代码示例是否完整、可理解 |
| 动静态覆盖 | 15% | 15 | 动态工具链和静态工具链是否均覆盖 |
| 结构规范性 | 10% | 10 | 是否遵循文档模板结构（功能概述/源码/代码/差异） |
| 动静态差异说明 | 5% | 5 | 动静态差异表是否存在且准确 |
| 可维护性 | 5% | 5 | 是否存在冗余信息、标题层级是否一致 |

## 评分标准

### 1. 源码准确性（25 分）

| 得分 | 标准 |
|---|---|
| 25 | 所有源码参考位置路径真实存在、行号在合理范围内、函数名与源码一致 |
| 20 | 源码参考位置存在但部分行号有偏差（±5 行以内） |
| 15 | 源码参考位置存在但行号偏差较大（±10 行以内）或函数名有小错误 |
| 10 | 部分源码参考位置路径不存在或明显错误 |
| 5 | 大部分源码参考位置路径不存在 |
| 0 | 无源码参考位置或全部错误 |

### 2. 内容完整性（20 分）

| 得分 | 标准 |
|---|---|
| 20 | 覆盖该主题的所有关键规则/转换分支/校验逻辑，无遗漏 |
| 16 | 覆盖主要规则但存在 1-2 个次要分支遗漏 |
| 12 | 覆盖核心规则但遗漏较多次要分支 |
| 8 | 仅覆盖基本规则，关键分支遗漏 |
| 4 | 内容严重不完整，仅覆盖皮毛 |
| 0 | 无实质内容 |

### 3. 代码示例质量（20 分）

| 得分 | 标准 |
|---|---|
| 20 | 转换前/后代码示例完整（含 Legacy 和 PU 模式），代码可直接对照理解转换逻辑 |
| 16 | 有转换前/后代码但缺少 Legacy 或 PU 之一 |
| 12 | 有代码示例但不完整（缺少关键参数或输出） |
| 8 | 有简短代码片段但不足以理解转换逻辑 |
| 4 | 仅有一行代码示意 |
| 0 | 无代码示例 |

### 4. 动静态覆盖（15 分）

| 得分 | 标准 |
|---|---|
| 15 | 动态工具链和静态工具链的转换/校验逻辑均完整覆盖 |
| 12 | 动静态均覆盖但静态部分较浅 |
| 9 | 仅覆盖动态工具链，静态工具链缺失或仅一句话提及 |
| 6 | 仅提及动态工具链存在 |
| 3 | 未区分动静态 |
| 0 | 无动静态区分 |

### 5. 结构规范性（10 分）

| 得分 | 标准 |
|---|---|
| 10 | 完全遵循文档模板（功能概述→动态→静态→动静态差异说明），子标题层级一致 |
| 8 | 基本遵循模板但子标题层级有 1-2 处不一致 |
| 6 | 部分遵循模板，缺少 1-2 个标准章节 |
| 4 | 结构松散，缺少多个标准章节 |
| 2 | 仅有关键词罗列，无结构 |
| 0 | 无结构 |

### 6. 动静态差异说明（5 分）

| 得分 | 标准 |
|---|---|
| 5 | 有动静态差异对比表，维度准确且内容有实质对比 |
| 4 | 有差异说明但以文字描述而非表格 |
| 3 | 有差异说明但内容笼统 |
| 1 | 仅一句话提及差异 |
| 0 | 无动静态差异说明 |

### 7. 可维护性（5 分）

| 得分 | 标准 |
|---|---|
| 5 | 无冗余信息，标题层级一致，无重复内容 |
| 4 | 有少量冗余（1-2 处）但不影响理解 |
| 3 | 有明显冗余（3-5 处重复内容） |
| 2 | 冗余较多，同一信息在多处重复 |
| 1 | 严重冗余，大量重复 |
| 0 | 文档不可维护 |

## 评分流程

### 步骤 1：确定评估范围

读取 `docs/specs/01-编译转换/` 和 `docs/specs/02-校验规则/` 目录下的所有 `.md` 文件列表。

### 步骤 2：逐文档打分

对每个文档，按以下流程评估：

1. **读取文档内容**：用 Read 工具读取全文
2. **源码准确性验证**：
   - 提取文档中所有 `相对路径:行号` 格式的源码引用
   - 用 Read 工具抽样验证 3-5 个关键引用（验证路径是否存在、行号附近内容是否匹配函数名）
   - 按验证结果打分
3. **内容完整性评估**：
   - 对照源码中该主题的函数/分支数量
   - 统计文档覆盖的分支数量
   - 按覆盖率打分
4. **代码示例质量评估**：
   - 检查是否有转换前代码
   - 检查是否有转换后代码（Legacy 模式）
   - 检查是否有转换后代码（Partial Update 模式）
   - 检查代码是否包含关键参数和输出
   - 按完整度打分
5. **动静态覆盖评估**：
   - 检查是否有 `## 动态` 章节
   - 检查是否有 `## 静态` 章节
   - 检查静态章节是否有实质内容（非仅一句话）
   - 按覆盖度打分
6. **结构规范性评估**：
   - 检查是否包含标准章节：功能概述、动态、静态、动静态差异说明
   - 检查标题层级是否一致（`###` 不应跳到 `#####`）
   - 按符合度打分
7. **动静态差异说明评估**：
   - 检查是否有差异对比表
   - 检查表格维度是否准确
   - 按质量打分
8. **可维护性评估**：
   - 检查是否有同一信息在多处重复
   - 检查标题层级是否一致
   - 按冗余程度打分

### 步骤 3：汇总评分

计算加权总分，按以下等级分类：

| 等级 | 总分范围 | 说明 |
|---|---|---|
| A（优秀） | 90-100 | 文档质量高，可作为参考范例 |
| B（良好） | 75-89 | 文档质量良好，有小幅优化空间 |
| C（合格） | 60-74 | 文档基本可用，有明显优化需求 |
| D（不合格） | 0-59 | 文档质量差，需要重写或大幅补充 |

### 步骤 4：生成报告

输出格式：

```markdown
## 规格文档质量评估报告

### 汇总统计

| 等级 | 数量 | 占比 |
|---|---|---|
| A（90-100） | X | X% |
| B（75-89） | X | X% |
| C（60-74） | X | X% |
| D（0-59） | X | X% |

### 详细评分

| 文档 | 源码准确性(25) | 内容完整性(20) | 代码示例(20) | 动静态覆盖(15) | 结构规范(10) | 差异说明(5) | 可维护性(5) | 总分 | 等级 |
|---|---|---|---|---|---|---|---|---|---|
| 01-01 struct转换 | 25 | 18 | 20 | 15 | 10 | 5 | 5 | 98 | A |
| ... | ... | ... | ... | ... | ... | ... | ... | ... | ... |

### 低分文档优化建议

#### C 级文档

| 文档 | 主要失分项 | 优化建议 |
|---|---|---|
| XX-XX xxx.md | 源码准确性(10/25)：行号偏差大 | 重新核对源码行号 |
| ... | ... | ... |

#### D 级文档

| 文档 | 主要失分项 | 优化建议 |
|---|---|---|
| XX-XX xxx.md | 内容完整性(4/20)：仅覆盖皮毛 | 补充关键转换分支和代码示例 |
| ... | ... | ... |
```

## 特殊规则

### 编译转换文档（01-XX）额外检查项
- 如果文档涉及装饰器转换，检查是否同时覆盖动态工具链（`process_component_member.ts` 等）和静态工具链（`property-translators/` 对应文件）
- 如果文档涉及组件转换，检查是否有转换前/后代码对照
- 附录类文档（交叉验证）不要求代码示例，代码示例维度自动按 15 分满分折算

### 校验规则文档（02-XX）额外检查项
- 检查是否包含报错信息（动态和静态各自的文案）
- 检查是否包含错误码
- 检查是否有反例和正例代码
- 校验规则文档不要求 Legacy/PU 区分，代码示例维度按"是否有反例+正例"评分

### 合并类文档（如 01-09 组件转换）额外检查项
- 检查各子章节结构是否统一（每个特殊组件节是否都有：源码参考位置→转换前→转换后→关键逻辑）
- 检查是否有冗余信息（同一规则在多处重复）
- 可维护性维度权重提升至 10 分（从总分中额外扣除 5 分用于惩罚结构混乱）

## 使用示例

```
用户：对当前规格文档进行质量评估

执行流程：
1. 列出 docs/specs/ 下所有 .md 文件
2. 逐个读取并按 7 个维度打分
3. 抽样验证源码引用准确性（每个文档验证 3-5 个引用）
4. 汇总生成评估报告
5. 对 C/D 级文档给出具体优化建议
```
